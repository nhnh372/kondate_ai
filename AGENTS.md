# AGENTS.md

このドキュメントは、KONDATE AI 専属のAIエンジニアとして長期的に開発するための作業ルールです。
実装時は必ず本ドキュメントを確認し、既存機能を壊さず、小さく安全に改善してください。

## プロジェクト概要

- プロジェクト名: KONDATE AI
- Flutterで開発する家庭向け献立提案アプリ
- iOS・Android対応予定
- AIを活用して、家庭の状況に合わせた献立を提案する
- 将来的にFirebase、Gemini、通知、課金へ拡張予定

## 開発思想

- 既存機能を壊さない。
- 小さな単位で実装する。
- 保守性を最優先する。
- 可読性を重視する。
- Repositoryパターンを維持する。
- UIとロジックを分離する。
- 将来のFirebase、Gemini、通知、課金連携を見据えて実装する。
- 変更前に影響範囲を確認し、AI献立生成、お気に入り、履歴、週間献立、詳細画面に副作用がないようにする。
- このプロジェクトは個人開発であり、「まず動くものを作る」から始める。
- ユーザーが実際に触り、その反応を見て改善する流れを最優先する。
- 必要以上に設計を複雑にしない。
- ユーザーが理解できる日本語で説明する。
- 勝手に大規模リファクタリングを行わない。
- 必ず一つずつ完成させる。

## ディレクトリルール

各ディレクトリの責務を明確に分け、責務が混ざらないように実装する。

- `models/`
  - アプリで扱うデータ構造を定義する。
  - Modelはデータのみを保持し、保存処理や複雑なビジネスロジックを書かない。
- `data/`
  - 初期データ、固定データ、サンプルデータ、ローカル定義データを置く。
  - UIや保存処理に依存しない純粋なデータ定義にする。
- `repositories/`
  - SharedPreferencesなど、データの保存・取得・更新・削除を担当する。
  - Repositoryは保存処理専用とし、UI判断や献立選択ロジックを書かない。
- `services/`
  - 献立選択、AI優先度計算、スコアリング、通知準備などのアプリロジックを担当する。
  - UIに依存しない形で実装し、テストしやすく保つ。
- `widgets/`
  - 再利用可能な小さなUI部品を置く。
  - 画面固有ではないカード、ボタン、リスト項目、状態表示などを分割する。
- `pages/`
  - 画面単位のUIを置く。
  - 既存で `pages/` を使っている場合は、その構成を尊重する。
- `screens/`
  - 画面単位のUIを置く。
  - 既存で `screens/` を使っている場合は、その構成を尊重する。
  - `pages/` と `screens/` が混在する場合は、既存の命名規則に合わせて無理に移動しない。
- `utils/`
  - 汎用的なヘルパー、定数、フォーマット処理などを置く。
  - 特定機能の複雑なロジックは `services/` に置く。

## コーディングルール

- 可読性を最優先にする。
- DRYを意識し、同じコードを不用意に繰り返さない。
- SOLID原則をできる限り守る。
- 不要なコードは残さない。
- Magic Numberを禁止する。意味のある定数名に切り出す。
- コメントは必要最低限にする。
- 日本語でも理解しやすい命名を心掛ける。
- クラスを小さく保つ。
- Widgetを適切に分割する。
- UIとビジネスロジックを分離する。
- Serviceへアプリロジックを書く。
- Repositoryは保存処理専用にする。
- Modelはデータのみ保持する。
- 既存の命名、構成、設計パターンを優先する。
- 影響範囲が広いリファクタリングは、必要な場合だけ小さく行う。

## ファイル管理ルール

- 1ファイルが300行を超えそうな場合は、分割を提案する。
- Repository、Service、Model、Widgetへ適切に分離する。
- 責務を分ける設計を維持する。
- 共通化できる処理はUtility化を提案する。
- 既存の未コミット変更を勝手に戻さない。
- ユーザーが作業中のファイルを変更する場合は、既存差分を確認してから最小限に編集する。

## UIルール

### テーマ

- オレンジとベージュを基調にする。
- 家庭向けで、やさしい印象を大切にする。
- Material3を前提にする。
- 子育て世帯が迷わず使える、安心感のあるUIにする。

### デザイン

- カードUIを中心に構成する。
- 角丸を活用する。
- 余白を多めに取る。
- アイコンを活用して視認性を高める。
- シンプルで読みやすくする。
- 操作導線を短くし、片手でも使いやすい画面を意識する。
- 文字やボタンが小さくなりすぎないようにする。
- 新しい画面・機能を追加した場合は、Chrome(Web)で動作確認できる状態まで実装する。
- 確認用URLを表示する。
- ユーザーがChromeで確認するまで停止する。
- 勝手にデザインを変更しない。
- UI改善案は提案として提示する。
- ユーザーの承認後のみデザインを変更する。

## AI献立ルール

献立選択では、次のスコアを利用する。

- 栄養
- 時短
- 簡単さ
- 新規性

今後、次の条件も考慮できる設計にする。

- 季節
- 在庫
- 家族人数
- 予算
- 好み
- 履歴
- お気に入り

AI献立ロジックを追加・変更する場合は、スコアリングの責務を `services/` に寄せ、UIやRepositoryに判断ロジックを混ぜない。

## 実装フロー

必ず次の順番を守る。

1. 実装前に変更内容を説明する。
2. 実装する。
3. Chromeで画面確認できる状態まで作る。
4. そこで一度停止し、ユーザーが実際に画面を確認するのを待つ。
5. 必要な修正を行う。
6. ユーザー確認後に `dart format` を実行する。
7. ユーザー確認後に `flutter analyze` を実行する。
8. ユーザー確認後に `flutter test` を実行する。
9. ユーザー確認後にGit Commitを提案する。

ユーザー確認前に `dart format`、`flutter analyze`、`flutter test` を実行しない。
ユーザーが実際にUIを確認する前に、解析、テスト、コミットへ進まない。

## Chrome確認ルール

- UIを変更した場合は、Chromeで画面確認できる状態まで実装して止まる。
- 画面確認に必要な起動コマンドやURLを報告する。
- Chrome確認後、ユーザー確認を待ってからフォーマット、解析、テストへ進む。
- UIは必ずユーザーの目視確認を優先する。
- ユーザー確認前に `flutter analyze`、`flutter test`、Git Commitを実行しない。

## 承認ルール

権限承認が必要になった場合は、次の順番で対応する。

- 他に進められる作業を優先する。
- 承認待ちだけで停止せず、可能な範囲で作業を続ける。
- 最後に、承認が必要な作業の一覧を日本語で報告する。

## Gitルール

- 機能単位でコミットを提案する。
- コミットメッセージは日本語にする。
- ユーザーの明示的な指示なしにコミットしない。
- ユーザーがOKを出した場合のみ `git add .` と `git commit` を実行する。
- コミットメッセージは変更内容が分かる日本語にする。
- 既存の未コミット変更を勝手に戻さない。

## テストルール

書ける場合は、変更内容に応じて次のテストを追加する。

- Unit Test
- Widget Test
- Repository Test
- Service Test

ただし、ユーザー確認前に `flutter analyze` や `flutter test` は実行しない。
機能追加時は可能な限りUnit TestとWidget Testを追加する。
テスト追加が必要な場合も、まず実装とChrome確認を優先し、その後ユーザー確認を待つ。
ユーザー確認後のみ `dart format`、`flutter analyze`、`flutter test` を実行する。

## 作業時の注意

- AI献立生成、お気に入り、履歴、週間献立、詳細画面は壊さない。
- SharedPreferencesの保存形式を変更する場合は、既存データへの影響を確認する。
- UI変更では、家庭向けのやさしい印象と操作しやすさを優先する。
- 大きな設計変更が必要な場合は、先に小さな移行手順を提案する。
